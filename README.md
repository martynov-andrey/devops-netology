#### 1. Какого типа команда `cd`? Попробуйте объяснить, почему она именно такого типа; опишите ход своих мыслей, если считаете что она могла бы быть другого типа.

`cd` - это встроенная команда оболочки (sh/bash/...), не отдельная программа или системный вызов. Команда меняет состояние самой оболочки.
Каждый процесс имеет свой текущий рабочий каталог. Если бы это была внешняя команда, исполняемый файл создал бы процесс со своим собственным рабочим каталогом, не влияя на тот из текущей оболочки. Даже если внешняя команда изменит свой каталог, это изменение исчезнет при выходе из внешнего процесса.

#### 7. Выполните команду `bash 5>&1`. К чему она приведет? Что будет, если вы выполните `echo netology > /proc/$$/fd/5`? Почему так происходит?

Команда `bash 5>&1` - создаст файловый дескриптор с номером 5 по следующему пути `/proc/$$/fd/5` и перенаправит его в stdout:

```bash
    $ ls -l /proc/$$/fd/5
    lrwx------ 1 vagrant vagrant 64 Nov 27 10:03 /proc/1429/fd/5 -> /dev/pts/0
    $ echo netology > /proc/$$/fd/5
    netology
```

Все что будет записано в созданный и перенаправленный в стандартный поток вывода файловый дескриптор будет выведено на экране.

#### 8. Получится ли в качестве входного потока для pipe использовать только stderr команды, не потеряв при этом отображение stdout на pty? Напоминаем: по умолчанию через pipe передается только stdout команды слева от `|` на stdin команды справа. Это можно сделать, поменяв стандартные потоки местами через промежуточный новый дескриптор, который вы научились создавать в предыдущем вопросе.

```bash
    $ ls -l ~/ /tmp1 6>&2 2>&1 1>&6 | grep cannot -c
    /home/vagrant/:
    total 4
    -rw-rw-r-- 1 vagrant vagrant 16 Nov 27 08:56 test.txt
    1
```

`6>&2` - создан дескриптор и перенаправлен в поток ошибок (stderr);

`2>&1` - stderr перенаправлен в выходной поток (stdout);

`1>&6` - stdout перенаправлен в новый дескриптор.

#### 12. При открытии нового окна терминала и `vagrant ssh` создается новая сессия и выделяется pty. Это можно подтвердить командой `tty`, которая упоминалась в лекции 3.2. Однако:

     ```bash
     vagrant@netology1:~$ ssh localhost 'tty'
     not a tty
     ```

     Почитайте, почему так происходит, и как изменить поведение.

```bash
      $ ssh -t localhost 'tty'
      vagrant@localhost's password:
      /dev/pts/1
      Connection to localhost closed.
```

Ответ: При подключении к удаленному серверу по ssh терминал tty сразу не выделяется и shell не создается, управление передается программе login, ожидается авторизации пользователя.
Опция -t принудительно выделяет псевдотерминал для выполнения программ и отображения вывода на удаленном компьютере.

#### 13. Бывает, что есть необходимость переместить запущенный процесс из одной сессии в другую. Попробуйте сделать это, воспользовавшись `reptyr`. Например, так можно перенести в `screen` процесс, который вы запустили по ошибке в обычной SSH-сессии.

Терминал с запущенным процессом:

```bash
      $ tty
      /dev/tty1
      $ top
```

Терминал куда будем перехватывать процесс и помещать в сессию `screen`:

```bash
      $ screen -S 
      $ tty
      /dev/pts/1
      $ ps -A | grep tty1 | grep top
      966 tty1     00:00:00 top
      $ reptyr 966
      Ctrl-a d
      $ screen -ls
      There is a screen on:
        895.top (11/28/2021 07:13:44 PM)        (Detached)
      1 Socket in /run/screen/S-vagrant.
      $ ps aux | grep top
      vagrant      895  0.1  0.2   9848  2952 ?        Ss   19:13   0:00 SCREEN -S top
      vagrant      966  1.3  0.3  11848  3892 pts/2    Ss+  19:20   0:04 top
      vagrant      976  0.0  0.0      0     0 tty1     Z    19:21   0:00 [top] <defunct>
      vagrant      985  0.0  0.0   8900   736 pts/0    R+   19:25   0:00 grep --color=auto top
```

Для выполнения задания необходимо установить программу `reptyr` и отредактировать в файле `/etc/sysctl.d/10-ptrace.conf` строку `kernel.yama.ptrace_scope = 0` с перезапуском виртуальной машины.   

#### 14. `sudo echo string > /root/new_file` не даст выполнить перенаправление под обычным пользователем, так как перенаправлением занимается процесс shell'а, который запущен без `sudo` под вашим пользователем. Для решения данной проблемы можно использовать конструкцию `echo string | sudo tee /root/new_file`. Узнайте что делает команда `tee` и почему в отличие от `sudo echo` команда с `sudo tee` будет работать.

Ответ: 

Команда `tee` читает из из стандартного потока ввода (stdin) и одновременно пишет в стандартны поток вывода (stdout) и в файл указаный в качестве параметра. В примере команда получает поток вывода команды `echo` перенаправленный через pipe, `sudo tee` будет работать.
